# FolderVault

> "아빠가 폴더 비밀번호 거는 프로그램 하나 만들어 달라고 하셨다. 좀 과하게 만들어 버렸다."

폴더를 실제로 암호화해서 보관하는 데스크톱 프로그램입니다. "숨기기"가 아니라
파일 내용 자체를 암호화하므로, 올바른 비밀번호 없이는 누구도(만든 사람
포함) 데이터를 복원할 수 없습니다.

리뷰받고 깨져 보려고 공개합니다. 그게 공개하는 이유입니다.

English: [README.md](README.md)

## 실행 방법

1. `start.bat` 더블클릭
   - 최초 1회만 자동으로 환경을 설정합니다(인터넷 필요, 1~2분).
   - 이후에는 즉시 실행됩니다.

수동 실행:

```
pip install -r requirements.txt
python folder_vault.py      # GUI (설정에서 한국어/English 전환)
python test_vault.py        # 자가검증 143개(25개 카테고리)
```

## 사용법

| 버튼 | 기능 |
|------|------|
| 폴더 잠그기 | 폴더 선택 → 비밀번호 설정 → 암호화. 잠그기 직전에 원본 처리 방식(영구 삭제 / 원본 유지 / 취소)을 직접 선택하며, 삭제 선택 시 경로를 보여주는 최종 경고가 한 번 더 표시됩니다 |
| 보안 폴더 열기 | `.foldervault` 선택 → 비밀번호 입력 → 원하는 위치에 폴더 복원 |
| 다시 잠그기 | 복원해서 작업한 폴더를 다시 암호화 |
| 비밀번호 변경 | 기존 비밀번호 → 새 비밀번호로 재암호화 |
| 설정 | 언어(한국어/English), 암호화 강도, 원본 삭제 방식 |

폴더별로 다른 비밀번호를 설정할 수 있고, 그 폴더를 열려면 반드시 해당
비밀번호를 입력해야 합니다.

## 보안 사양

| 항목 | 사용 기술 |
|------|-----------|
| 비밀번호 → 키 유도 | Argon2id(메모리 하드, 표준 256MB / 강력 512MB / 편집증 1GB) |
| 키 분리 | HKDF-SHA256(헤더 / 인덱스 / 파일별 독립 서브키) |
| 데이터 암호화 | 캐스케이드 AES-256-GCM → ChaCha20-Poly1305(독립 키·논스 2중 인증) |
| 페퍼 모드 | 앱 내장(기본·이식 가능) 또는 OS 키체인(Windows DPAPI, 이 PC·계정 전용). 옵트인 + 백업/복원 |
| 무결성/진본 | 청크별 2중 AEAD + 볼트 전체 하이브리드 서명 Ed25519 + ML-DSA-65(FIPS 204) — 둘 다 검증해야 열림 |
| 양자 내성 | 기밀성은 이미 PQ 안전(대칭·비밀번호 기반). 무결성에 ML-DSA-65 적용 |
| 메타데이터 | 파일명·구조 암호화 + Padmé 패딩(파일 크기 은닉, 낭비 < ~12%) |
| 변조 방지 | 청크별 AAD 바인딩(경로·순번·총개수) → 재정렬·잘림·교체 차단 |
| 큰 파일 | 1 MiB 청크 스트리밍(수 GB 파일도 메모리 부담 없이) |

한 번 잠그면 PC를 껐다 켜도, 이 프로그램을 삭제해도, OS를 재설치해도
`.foldervault` 파일만 있으면 복원할 수 있고, 비밀번호 없이는 영구히
복호화할 수 없습니다. 보호는 백그라운드 프로세스가 아니라 암호화 그
자체로 보장됩니다.

## 양자 내성을 왜 절반만 적용했는가

이 볼트의 기밀성은 이미 양자 안전입니다. 공개키 키교환이 없는 순수
대칭 암호(비밀번호 → Argon2id → AES-256 + ChaCha20)라서 "지금
수확해서 나중에 복호" 위협이 적용되지 않고, Shor 알고리즘도 무력하며
Grover만 영향을 주는데 그것도 256비트를 절반으로 줄일 뿐 여전히
현실적으로 불가능합니다.

그래서 ML-KEM은 일부러 넣지 않았습니다. 보호할 키교환이 없는데
붙이는 건 보안 연극입니다. 양자 컴퓨터가 실제로 의미 있는 곳은 무결성
서명뿐이라, 거기에만 ML-DSA-65를 고전 Ed25519와 하이브리드로
적용했습니다(둘 다 통과해야 열림). 과장 없이 정확히 적용한 범위입니다.

## 데이터 보존 안전장치

원본을 삭제하는 프로그램이라, 데이터 손실을 막는 것을 가장 우선합니다.

- 삭제 전 전체 검증: 임시 파일에 쓴 뒤 모든 청크를 실제로 복호화하고
  구조 정합성까지 확인합니다. 통과한 경우에만 볼트가 배치되고 원본
  삭제가 진행됩니다. 한 조각이라도 실패하면 원본은 절대 삭제되지
  않습니다.
- TOCTOU 차단: 실제 청크 수·크기·파일 ID를 인증된 인덱스에 기록해
  스캔과 암호화 사이에 파일이 바뀌어도 정확한 비밀번호로 항상 열립니다.
- 볼트 자기삭제 방지: 보안 파일을 암호화 대상 폴더 안에 저장하려 하면
  거부합니다.
- 비밀번호 변경 무손상: 평문을 디스크에 전혀 쓰지 않고 청크 단위로
  재암호화, 새 볼트가 전체 검증을 통과한 뒤에만 원자적으로 교체합니다.
- 링크/정션 차단: 심볼릭 링크/정션이 있으면 쓰기·삭제 전에 중단하고
  알려 줍니다.
- 검증된 것만 정확히 삭제: 라이브 폴더를 통째로 지우지 않고
  암호화·검증된 항목만 지웁니다.
- 정직한 보고: 삭제 실패 시 "완료"라고 거짓 보고하지 않고 평문이 남은
  항목을 정확히 알립니다.
- 변조 볼트 메모리 폭발(DoS) 차단, 콘솔 없는 실행에서도 조용한 실패
  없음(`crash.log` 기록).

## 정직하게 밝히는 한계

소프트웨어로 해결할 수 없는 진짜 한계입니다. 숨기는 것보다 밝히는 게
낫다고 생각합니다.

- 메모리상의 비밀번호/키: 파이썬의 immutable `str`, Tk 내부 사본,
  라이브러리가 반환하는 키 바이트는 순수 파이썬에서 신뢰성 있게
  지울 수 없습니다. "메모리 와이프" 흉내는 동작하지 않는 연극이라
  넣지 않았습니다. 실행 중 메모리 스캔·코어 덤프 위협에는 OS 수준
  보호(전체 디스크 암호화, 신뢰된 단독 사용 환경)가 필요합니다.
- 키체인 모드의 이식성 상실: OS 키체인 페퍼로 만든 볼트는 그 Windows
  계정·PC에서만 열립니다. 그래서 기본은 앱 모드이고, 키체인은 옵트인
  + 반드시 페퍼 백업입니다.
- 앱 내장 페퍼는 소스 공개에 대한 비밀이 아닙니다: 오픈소스라
  내장 페퍼는 보입니다. "`.foldervault` 파일만" 유출된 경우의
  추가 방어선일 뿐이며, 일부러 그렇게 설계했습니다. 페퍼를 바꾸거나
  지울 수 없게 한 이유는, 바꾸는 순간 기존 볼트가 전부 영구히 안
  열려서(데이터 손실) 그게 최악이기 때문입니다.
- 비밀번호/파일 분실 = 복구 불가. 설계상 의도이며 백도어·복구키가
  없습니다. `.foldervault` 파일은 반드시 백업하세요.
- SSD 안전 삭제 한계: 웨어 레벨링 때문에 어떤 소프트웨어도 원본
  흔적을 100% 제거한다고 보장할 수 없습니다. 실질적 보호는 삭제가
  아니라 암호화가 담당합니다.

## 어떻게 만들었나

AI 도움을 받아 만들었고, 정확히 어떻게 받았는지 밝힙니다.

- 구조, 위협 모델, 설계 결정은 제가 했습니다. 왜 PQC를 어디에만
  적용하는지, 왜 캐스케이드 AEAD인지, 왜 Padmé 패딩인지, 무엇을
  테스트할지, 데이터 손실 안전 모델 — 전부 제 판단입니다.
- 암호 프리미티브 연결 코드는 제 사양서로 Claude Opus 4.7가
  생성했습니다. 전부 검증된 라이브러리(`cryptography`,
  `argon2-cffi`)를 쓰고, 직접 만든 암호는 없습니다(의도).
- 통합·테스트·디버깅은 제가 했습니다. 25개 카테고리 143개 검증.

저는 14살이고 암호학을 아직 깊이 배우는 중입니다. 외부 리뷰를
받으려고 공개합니다. 시스템 수준의 결정은 제 것이고 구현은 AI 도움을
받은 것 — 이게 합리적이고 정직한 개발 방식이라고 생각하고, 숨기기보다
그냥 밝히는 쪽을 택했습니다.
